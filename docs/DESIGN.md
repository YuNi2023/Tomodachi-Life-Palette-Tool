# すぽいと帳 — 設計メモ

このドキュメントは、すぽいと帳（Tomodachi-Life-Palette-Tool）の設計判断をまとめた
ノートです。何を選んで、何を選ばなかったか、なぜそうしたか。

---

## TL;DR

**ブラウザ単体で完結する、フレームワークレスのキャンバス職人ツール。**

トモコレの 256×256 ドット絵と 84 色パレット制約に任意の画像を変換して、その色番号を
並べた塗り絵まで生成する。やってることはそれだけ。それを **ちゃんと作る** ために
意図的にやらなかったことが、このプロジェクトの輪郭。

---

## 設計判断の優先順位

1. **ローカルファースト** — 画像はサーバーに送らない。ブラウザ内で完結する
2. **デプロイの単純さ** — GitHub Pages に push したら動く
3. **コードの透明性** — ソースを開けば、書いた通りに動く（ビルドステップなし）
4. **多言語サポート** — ファン向けツールとして体験を分け隔てない

この 4 つを犠牲にしない範囲で機能を増やしていく。逆に言えば、これらを揺るがすような
変更は入れない。

---

## アーキテクチャ全体像

```
ブラウザ
└─ index.html
   └─ <script src=…> 直列ロード
      ├─ js/core/        状態・パレット・i18n の基盤
      ├─ js/canvas/      描画レイヤー（pixel canvas / overlay canvas）
      └─ js/features/    機能ごとのファイル
```

クラスも ES Modules も使わない。各 `.js` ファイルのトップレベル `function` が
グローバルになり、他ファイルから素直に呼べる。状態は `let imgData` `let viewMode`
`let gridSize` のようなモジュールスコープ変数。

このサイズのアプリなら、これで十分読めるし書ける。

### なぜフレームワークを使わないか

- React/Vue/Svelte の差別化要素を出せるほどのインタラクション複雑度がない
- バンドラ依存はビルド時間・デバッグ難度・依存性の更新コストを連れてくる
- 「F12 を開けば書いたソースが動いている」という透明性が好き

機能が増えて状態管理が複雑化したら、その時に考える。今はまだ閾値の手前。

### なぜ手続き型・グローバル名前空間か

`new ColorPicker()` のような書き方は、**インスタンスが 1 個しか存在しない UI に対しては
無意味**。OOP は手段であって目的じゃない。

純粋関数で書ける部分（`findNearestPaletteIdx`, 平均化, パレットマッチ）はそうしている。
それで足りる。

## ファイル分割 — レイヤーではなく機能フォルダ

クリーンアーキテクチャ的な Domain / UseCase / Infrastructure の層分けはしない。
代わりに **機能のかたまり** で切る。

```
js/features/crop-tool.js     — クロップ機能ぜんぶ
js/features/view-mode.js     — 変換ビューぜんぶ（変換・保存・PBN生成）
js/features/color-info.js    — 色詳細パネル
js/features/recipe.js        — レシピ表示
…
css/features/*.css           — 同名の CSS ファイルが対応
```

クロップを直したい → `js/features/crop-tool.js` と `css/features/crop.css` を開く。
それで終わり。Feature-Sliced Design に近い思想で、規模が小さいほど効く方針。

---

## i18n — コードじゃなくデータ

```
js/i18n/
├─ ja.json
├─ en.json
├─ ko.json
└─ …  (12 言語)
```

DOM 側に `data-i18n="view.savePng"` と書いて、起動時に JSON から置換する。
新言語追加 = JSON ファイル 1 個足すだけ、JS コードは触らない。

CJK の改行不能文字対応として `data-i18n-html` も用意。`<wbr>` 入りの文字列を
innerHTML で挿入することで「ドット絵を / 保存」のような意味境界での折り返しを実現。

## インフラ — そもそも何もない

| 項目                | 有無 | 備考                                   |
| ------------------- | ---- | -------------------------------------- |
| バックエンド        | ❌   | サーバーレスではなく「サーバーがない」 |
| データベース        | ❌   | 永続化は LocalStorage に最小限         |
| 認証                | ❌   | 登録の摩擦を作りたくない               |
| アナリティクス      | ❌   | ユーザー行動は取らない                 |
| 第三者 CDN          | ❌   | 全アセットがリポジトリ内               |
| ビルド / CI         | ❌   | `git push` = デプロイ                  |
| サードパーティ API  | ❌   | fetch するのは同梱の i18n JSON だけ    |

「無い」を意図的に並べている。これがこのプロジェクトの輪郭。

---

## 「やらないこと」がプロダクト設計

- **画像を送信しない** — すべての処理がブラウザ内
- **アカウントを作らせない** — URL を開けば即使える
- **追跡しない** — cookie / storage は UI 設定保存のみ
- **広告を出さない** — そういうコンセプトじゃない
- **データを勝手に保持しない** — タブを閉じたら消える。共有したいときだけ明示的にエクスポート

これは消極的な制約じゃなくて、**プロダクトの個性**そのもの。

---

## 例外的に「持つ」ものたち

- **LocalStorage**: ルーラー / グリッド表示のトグル状態（UI 設定のみ）
- **Canvas サイズ検出キャッシュ**: 初回 PBN 保存時に 5000×5000 のテストキャンバスを作って
  iOS Safari 互換かどうか判定し、結果を `_maxSafeCanvasDim` にメモ化
- **Object URL**: ダウンロード用に一時的に作成、30 秒後に `URL.revokeObjectURL`

どれも「最小限・揮発的・他人と共有されない」を守る。

---

## ブラウザ互換の取り扱い

iOS Safari の canvas 4096px 制限のように、特定環境固有の制約がある場合は
**ランタイムで feature-detect する** スタンス。User-Agent 文字列で分岐するのは
脆いので避け、実際に小さなテストを走らせて挙動を確認する。

例:

```js
function _detectSafeCanvasDim() {
  // 5000×5000 を試して、両端のピクセルがちゃんと描けるか確認
  // 描けたらデスクトップ扱い、ダメなら iOS など制限環境扱い
}
```

これによって、デスクトップ / Android Chrome では高解像度の番号塗り絵を生成し、
iOS では破綻しないサイズに自動調整される。

## 機能を追加するときの指針

将来の自分（または PR を送ってくれる人）向け。新機能を入れるたび、これを守る:

- **既存スタイルを踏襲** — クラス・ES Modules・新パターンを安易に持ち込まない
- 関数は機能別ファイル（`js/features/<機能名>.js`）にグローバル名前空間で追加
- 状態は既存の `let xxx = ...` パターンを踏襲。必要なら同じファイルにモジュール
  スコープ変数を増やす
- 環境依存の判定（iOS canvas 上限など）は **遅延評価 + メモ化** で書く。クラスや
  オブザーバーパターンを持ち込む必要はない
- LocalStorage はユーザー設定の最小限まで。作業データは保存しない
- 多言語対応が必要な UI は `js/i18n/ja.json` をマスターとして全 12 言語に追加
- CSS は機能と 1:1 対応の `css/features/*.css` に書く

---

## トレードオフを正直に書いておく

- **テスタビリティが低い** — グローバル状態 + Canvas 出力で自動テストを書きづらい。
  目視で確認している。本格的に育てるならここを直すべき
- **大規模化に弱い** — 複数開発者が入ったらグローバル汚染が事故になる。
  今のところ一人で書いてるので問題が顕在化していない
- **状態の関係性が暗黙** — `viewMode` `gridSize` `imgData` `convertedData` の依存
  関係はコード読まないとわからない。だからこのドキュメントを書いた

---

## 一言で

ブラウザだけで動く、自分で書いて自分でデプロイできる、必要な機能だけが入った道具。

> 「これ以上増やさない」と決めることのほうが、機能を増やすことより難しい。
